# Website Revision Patterns

## Purpose
This artifact turns critique into concrete upgrade moves. It exists so a weak website direction can be revised through specific changes to copy, hierarchy, section structure, pacing, and visual contrast instead of vague instructions to "make it better."

Use it after the anti-slop critique rubric identifies a failure pattern. The rubric names what is wrong; this artifact names the revision move that should change the output.

Use `runtime/web-quality/taste-calibration.md` alongside this file when choosing between plausible revision directions. Revision patterns say how to change the work; taste calibration sets the premium bar for which change is actually worth making.

## Before -> After Patterns
Use named transformations so revision work stays concrete and implementation-facing.

### Copy: Broad claim -> Specific promise
- **Before:** broad value statements, premium-sounding filler, or claims that could fit any product.
- **After:** a product-specific promise with a named user, concrete outcome, clearer mechanism, or real constraint.
- **Use when:** critique finds vague positioning or cliché copy.

### Hierarchy: Flat stack -> Decisive lead
- **Before:** hero, subhead, proof, and CTA carry similar weight, so nothing leads.
- **After:** one dominant idea leads, supporting proof arrives in the right order, and CTA emphasis follows the narrative instead of competing with it.
- **Use when:** critique finds limp hierarchy or weak narrative priority.

### Section structure: Default sequence -> Argument sequence
- **Before:** generic hero / features / testimonials / CTA order with no product-specific reason.
- **After:** sections are reordered into an argument that fits the product, audience, objections, and proof burden.
- **Use when:** critique finds generic structure or safe-but-forgettable layout decisions.

### Pacing: Even rhythm -> Controlled contrast
- **Before:** every section has similar density, width, tempo, and informational weight.
- **After:** dense and sparse moments alternate on purpose, key sections breathe, and repetition is broken so the page develops tension and release.
- **Use when:** critique finds flat visual rhythm or monotonous reading flow.

### Visual contrast: Uniform treatment -> Strategic emphasis
- **Before:** every block uses similar scale, contrast, spacing, and component treatment.
- **After:** important ideas receive stronger contrast through size, spacing, grouping, reduction, or composition, while less important material recedes.
- **Use when:** critique finds flat visual rhythm, buried proof, or a page that feels clean but forgettable.

## Pattern Selection Rules
Choose revision patterns based on the critique category and the visible failure signature.

- If the issue is **unclear meaning**, start with **Broad claim -> Specific promise**.
- If the issue is **weak page leadership**, start with **Flat stack -> Decisive lead**.
- If the issue is **generic flow or template sequencing**, start with **Default sequence -> Argument sequence**.
- If the issue is **monotony across sections**, start with **Even rhythm -> Controlled contrast**.
- If the issue is **nothing stands out visually**, start with **Uniform treatment -> Strategic emphasis**.

When multiple failures are present, revise in this order:
1. Copy and promise clarity
2. Hierarchy and lead idea
3. Section order and structure
4. Pacing across the page
5. Visual contrast and emphasis

Do not start with surface styling when the real problem is framing, sequencing, or meaning.

## Anti-Regression Rules
Revisions should not fall back into generic template output while trying to fix other problems.

- Do not replace weak copy with different premium-sounding filler.
- Do not add extra sections unless they strengthen the page argument.
- Do not increase visual intensity everywhere; contrast only works when some things stay quiet.
- Do not solve hierarchy problems by making every heading larger.
- Do not use references or style cues without translating them into concrete changes in copy, sequence, emphasis, or interaction.
- Re-check revised output against the anti-slop critique rubric so the fix does not simply move the failure into a different category.

## Constraint Notes
Revision patterns must respect the project posture established elsewhere.

- Keep the guidance local-first: prefer revision moves that can be applied directly in the artifact, prompt, or implementation without relying on remote scoring systems or external review services.
- Keep the guidance compatible with governed fallback: if a preferred revision move cannot be applied directly, fall back to the next explicit pattern instead of improvising vague polish language.
- Do not overstate certainty, automation, or evaluation maturity. These patterns guide human and agent revision work; they do not claim a live scoring engine.
