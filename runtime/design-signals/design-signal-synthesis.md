# Design Signal Synthesis

## Purpose
This artifact defines how gathered references become reusable signal outputs for later adaptation, critique, and implementation work.

It exists to convert bounded reference ingestion into a sharper design-language surface that can be reused without dragging raw inspiration, source noise, or imitation pressure into downstream tasks.

## Signal Card Shape
A synthesized output should be expressed as a signal card rather than a loose mood-board note.

Each signal card should include:

- **source class** — what kind of reference produced the signal, such as design article, site reference, component example, trend scan, or owner-provided reference
- **observed pattern** — the concrete structural, interaction, pacing, hierarchy, or presentation pattern actually observed
- **design implication** — what the pattern suggests for the current work, stated as a usable implementation-facing direction
- **anti-pattern risk** — how the signal could be misapplied, flattened into slop, or mistaken for permission to imitate
- **reuse note** — how the signal should be adapted, constrained, or combined when reused later

The card should stay small enough to be inspected quickly and specific enough to survive handoff.

## Translation Rules
References must be translated into guidance, not preserved as aesthetic authority.

Translation should favor reusable guidance such as:

- **structural guidance** — information architecture, layout logic, grouping, sequencing, navigation framing, or component relationships
- **narrative guidance** — voice posture, explanation rhythm, message ordering, reveal strategy, or emphasis logic
- **pacing guidance** — how dense or sparse sections feel, where compression helps, where pause or whitespace clarifies, and how the interface moves between beats
- **contrast guidance** — how importance is differentiated through scale, spacing, tone, interaction weight, or visual restraint

Translation should avoid copying surface styling, distinctive expression, or recognizable aesthetics just because they were present in the source.

A good synthesized signal explains what is reusable about the reference and why. It does not tell later work to recreate the source.

## Synthesis Quality Rules
A synthesized signal is useful only if it is implementation-facing and falsifiable enough to critique.

Useful signals should be:

- specific about what was observed
- clear about what is inference versus direct observation
- legible to later adaptation work without needing the original reference open beside it
- framed in terms of design consequence, not just taste language
- honest about limits, uncertainty, and failure modes
- reusable across contexts without becoming generic sludge

Weak synthesis usually sounds like vague mood-board language, trend worship, or empty style praise. If a signal could be reduced to "make it feel premium" or "more modern" without losing meaning, it is not sharp enough.

## Adaptation Boundary
Synthesis should feed later adaptation only through bounded design signals.

It should not hand downstream systems raw personal references, source-specific identity cues, or aesthetic instructions that collapse into memory retention or imitation.

Any adaptation layer that consumes synthesized signals should remain constrained by the owner-taste adaptation rules rather than treating synthesis as permission to accumulate personal memory.

## Non-Goals
Synthesis is not imitation.

Synthesis is not scraping for duplication.

Synthesis is not preserving raw references as downstream templates.

Synthesis is not blind trend-following.

Synthesis is not an excuse to launder copied aesthetics into implementation language.
